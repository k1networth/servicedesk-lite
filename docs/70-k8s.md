# Kubernetes

Helm chart реализован в `infra/k8s/helm/servicedesk-lite/`.
Локальный кластер — [kind](https://kind.sigs.k8s.io/) (Kubernetes in Docker).

## Быстрый старт

```bash
# Создать kind-кластер, установить ingress-nginx, задеплоить стек
make k8s-up

# Проверить
curl -X POST http://localhost:8080/tickets \
  -H 'Content-Type: application/json' \
  -d '{"title":"test","description":"k8s demo"}'
```

## Рабочие нагрузки

### Stateless сервисы (Deployment)

Все три сервиса имеют:

| Параметр | Значение |
|---|---|
| `readinessProbe` | HTTP `/readyz` — реальная проверка БД через `pg.PingContext` |
| `livenessProbe` | HTTP `/healthz` |
| `resources.requests` | cpu: 25–100m, memory: 64Mi |
| `resources.limits` | cpu: 500m–1000m, memory: 256Mi |
| `initContainers` | `wait-postgres` + `wait-kafka` (ticket-service) |

Конфигурация через ConfigMap + Secret (не хардкожено в образе).

### Stateful зависимости

**Postgres** (StatefulSet):
- Image: `postgres:16-alpine`
- PVC: `ReadWriteOnce`, 1Gi
- Tuning через `-c` флаги: `max_connections=200`, `shared_buffers=256MB`, `synchronous_commit=off`
- Readiness/Liveness: `pg_isready`

**Kafka** (StatefulSet, KRaft mode):
- Image: `apache/kafka:3.7.0`
- Headless Service для inter-broker + ClusterIP для клиентов
- Автоматическое создание топика при старте

### Migrations Job

При каждом `helm upgrade` запускается Job, который применяет SQL-миграции из `migrations/`.
Job использует init-контейнер `wait-postgres` перед запуском.

## Сетевая связность

```
Internet
  └─► Ingress (ingress-nginx, NodePort 8080)
        ├─► /           → ticket-service:8080
        ├─► /grafana    → grafana:80 (observability namespace)
        └─► /prometheus → prometheus:9090 (observability namespace)

ticket-service       → postgres:5432, kafka:9092
outbox-relay         → postgres:5432, kafka:9092
notification-service → postgres:5432, kafka:9092 (consumer)
```

## Масштабирование и надёжность

### HPA (Horizontal Pod Autoscaler)

Настроен для `ticket-service`:

```yaml
minReplicas: 1
maxReplicas: 3
targetCPUUtilizationPercentage: 70
```

При нагрузочном тесте (`hey -n 1000000 -c 200`) HPA поднимает до 3 реплик.

### PDB (Pod Disruption Budget)

```yaml
minAvailable: 1
```

Гарантирует доступность хотя бы одной реплики при rolling update или drain ноды.

### Rolling Update

- `maxUnavailable: 0` — нет даунтайма при обновлении
- `maxSurge: 1` — +1 под на время обновления
- readinessProbe гарантирует, что трафик идёт только на готовые поды

## Конфигурация (ConfigMap / Secret)

| Ключ | Тип | Описание |
|---|---|---|
| `APP_ENV` | ConfigMap | `dev` / `prod` |
| `KAFKA_BROKERS` | ConfigMap | headless DNS адрес Kafka |
| `KAFKA_TOPIC` | ConfigMap | название топика |
| `POSTGRES_DB` | ConfigMap | имя БД |
| `POSTGRES_USER` | ConfigMap | пользователь |
| `DB_MAX_OPEN_CONNS` | ConfigMap | размер пула соединений |
| `DB_MAX_IDLE_CONNS` | ConfigMap | idle соединения |
| `DATABASE_URL` | Secret | полный DSN с паролем |
| `POSTGRES_PASSWORD` | Secret | пароль Postgres |

## Observability в k8s

```bash
make k8s-obs-install   # kube-prometheus-stack в namespace observability
make k8s-obs-apply     # ServiceMonitors + Grafana dashboard
make k8s-obs-ui        # открыть Grafana в браузере
```

ServiceMonitor автоматически регистрирует все три сервиса в Prometheus.
Grafana dashboard: `http://localhost:8080/grafana` (admin/admin).

## kind-конфигурация

Файл: `infra/k8s/kind/kind-config.yaml`

- Один control-plane нод
- NodePort 8080 проброшен на хост (Ingress принимает HTTP трафик)
- Достаточно для демо и нагрузочного тестирования (1800+ RPS на 4 vCPU)

## Production отличия

| Параметр | kind (demo) | Production (Managed K8s) |
|---|---|---|
| Postgres | StatefulSet в кластере | Managed PostgreSQL |
| Kafka | StatefulSet в кластере | Managed Kafka |
| TLS | нет | cert-manager + Let's Encrypt |
| Секреты | Secret в кластере | Vault / KMS |
| Образы | локальная сборка | Container Registry + CI/CD |
| Узлы | 1 нод | multi-AZ node group |

Подробнее о production-топологии — [docs/80-iac.md](80-iac.md).
